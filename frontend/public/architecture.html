How the two voice paths actually work

WebSocket Browser ⟷ ElevenLabs Convai — used by Alex & Mila

  • Opened in-browser by @elevenlabs/client via Conversation.startSession({ agentId, connectionType: 'websocket' }).
  • Bidirectional. Uplink: mic PCM frames. Downlink: TTS audio chunks, transcripts (onMessage), mode changes (onModeChange), and client-tool invocations.
  • TTS audio is routed through a hidden <audio> element the SDK appends. We explicitly call .play() on it after startSession resolves to defeat Chrome's autoplay gating (fixes the 30-second first-token stall).
  • dynamicVariables: { visual_session_id } is passed at session start so the agent can forward it to any server tool it calls.

WebRTC/LiveKit Browser ⟷ HeyGen — used by Ann, Shawn, Katya, Graham

  • Frontend mints a LiveKit room token via POST /api/session/start (backend wraps HeyGen's /streaming/new + /streaming/start).
  • Browser joins the LiveKit room and receives the HeyGen avatar's rendered face + lipsync'd audio as standard WebRTC media tracks.
  • In LITE mode, HeyGen does only face rendering. STT/LLM/TTS is handled by a linked EL Convai agent referenced in elevenlabs_agent_config on the streaming session.
  • Because the return channel to the browser is audio+video only, EL client tools cannot fire browser code on this path — that's why HeyGen tutors need the SSE bridge for visuals.

server↔server HeyGen ⟷ ElevenLabs (internal, not reachable from browser)

  • HeyGen forwards inbound mic audio to EL for STT + LLM, receives TTS audio back, and muxes it into the lipsync rendering pipeline before shipping over LiveKit.
  • Dynamic variables set in elevenlabs_agent_config.dynamic_variables at session creation reach the EL agent over this link — which is how visual_session_id makes it into server-tool webhook bodies on the HeyGen path.

SSE Backend → Browser — the visual delivery bridge

  • Browser generates a UUID (visual_session_id) per mount, opens GET /api/visual/stream?sid=<uuid>. Backend stores Map<sid, Response>, sends 20-second pings.
  • When an EL server-tool render_visual webhook arrives at POST /api/visual/push, backend writes event: visual\ndata: {...spec}\n\n to the matching held-open Response.
  • Fallback: if sid is unknown, broadcast to every open client (hackathon-safe since one user at a time).
  • ⚠ Requires single-instance backend — Vercel's multi-instance serverless breaks the in-memory map. Demo uses cloudflared tunnel to a local process.

REST / webhook Everything else

  • Browser → Backend: /api/config, /api/session/start, /api/feedback.
  • EL → Backend: server-tool webhooks /api/visual/push, /api/tavily. Dynamic variables like visual_session_id are injected by EL into the POST body.
  • Backend → Tavily (grounding) + Gemini 2.5-flash (Teach grader) + HeyGen REST (session mint / credit balance).

client-tool Alex's fast path — no network hop for visuals

  • Alex alone has a client-type render_visual tool attached. When Alex invokes it, EL sends the call payload over the existing WebSocket; the SDK's clientTools.render_visual handler runs synchronously in the browser and pushes the spec straight to VisualRenderer via setVisual(spec).
  • Zero backend round-trip, zero SSE. This is why Alex's visuals land the moment he mentions them.
